TAAC 2026 / UNI-REC — EDA & 통합 백본 전략

대회: KDD Cup 2026 · Tencent UNI-REC Challenge 데이터: TAAC2026/data_sample_1000 생성: 2026-04-16
핵심 한 줄 요약 — UNI-REC은 “시퀀스 모델링과 피처 상호작용을 단일 동질 백본(homogeneous backbone)으로 통합”하라는 대회. 즉 SASRec-류와 DCN-류를 따로 두지 말고, 하나의 stackable unified block으로 둘 다 처리하라는 것이다. 주최측 권장 reading list가 OneTrans / InterFormer / HyFormer 셋이라는 점이 그 방향을 결정짓는다.

1. 대회 정보 (UNI-REC)

1.1 공식 task statement

“Towards Unifying Sequence Modeling and Feature Interaction for Large-scale Recommendation. … build a single, homogeneous backbone that handles both types of features and models their correlations with stackable, unified blocks.”
  • 대회: KDD Cup 2026 · Tencent UNI-REC Challenge (algo.qq.com).
  • 플랫폼: AngelML (주최측 학습/평가 리소스 제공 예정).
  • 데이터셋: privacy-preserving, de-identified industrial-scale dataset (Round 1·2 단계 공개).
  • 핵심 차별점: 시퀀스(behavior history)와 비-시퀀스(multi-field user/item/context features)를 분리된 모듈로 두지 말고 한 backbone으로 통합. unified tokenization이 사실상 강제된다.

1.3 주최측 권장 reading list 매우 강한 신호

주최측이 직접 추천한 세 논문은 UNI-REC가 정확히 어떤 방향을 원하는지를 결정짓는 가장 강한 단서다. 세 논문 모두 “시퀀스 + 피처 상호작용을 하나의 transformer/backbone에서 처리”하는 작업의 변종이다.

논문출판핵심 아이디어
OneTrans (arXiv 2510.26104) WWW 2026 (Tencent) S-tokens + NS-tokens 를 한 시퀀스로 concat → mixed causal attention + 파라미터 부분공유 + pyramid query pruning. 온라인 A/B에서 +5.68% GMV. UNI-REC의 가장 직접적인 청사진.
InterFormer (arXiv 2411.09852) CIKM 2025 (Meta) Interaction Arch + Sequence Arch + Cross Arch 의 3-block로 양방향 정보 흐름. early summarization 회피, 모드별 정보 보존.
HyFormer (arXiv 2601.12681) arXiv 2026 (ByteDance) “sequence vs feature 어느 쪽도 우선시 안 하는” hybrid attention. CTR 측면에서의 균형 재검토.
주의 — 2025년 baseline (baseline_2025: SASRec + RQ-VAE)은 UNI-REC의 baseline이 아니다. 그건 작년 “generative retrieval” task의 baseline이었고, 2026 UNI-REC는 task가 다르다(unified backbone for ranking-style). 2025 코드의 데이터로더만 차용하고 모델 본체는 위 3 논문 라인업 기반으로 새로 짜야 한다.

3. EDA 결과 + plots

모든 수치는 eda/out/{overview,deep,semantics}.json에서 가져왔으며, 재현은 eda/eda_0{1,2,3,4}_*.py로 가능합니다.

3.1 라벨 / 인터랙션 분포

1000명의 unique user × 1행씩 ⇒ 샘플은 user당 한 평가 케이스. label_type은 876 : 124 (87.6% / 12.4%) — 강한 클래스 불균형. 이건 next-item 생성 task가 아니라 per-row CTR/ranking task에 가깝다.

3.2 시퀀스 길이

도메인별 평균 시퀀스 길이는 A=701, B=571, C=449, D=1100. CDF를 보면 도메인 D는 1k+ 쪽으로 두꺼운 꼬리. 표준 SASRec/BERT4Rec window를 쉽게 초과한다.

3.3 도메인 vocab — 거의 disjoint

도메인 페어 Jaccard ≈ 0.7%–10%. 즉 4개 도메인은 서로 다른 ID 공간이며, 의미적으로도 서로 다른 신호 (예: 광고 노출 / 검색 쿼리 / 콘텐츠 토큰 / 앱 활동).

3.4 target item 멤버십 — Domain C가 가장 가까움

target item_id가 도메인 C vocab에 11.1% 등장, 나머지(A/B/D)는 ≤ 0.12%. 즉 도메인 C는 유사 아이템 노출/클릭 이력일 가능성이 높고, A/B/D는 보완 신호다.

3.5 (id, weight) 정렬 페어

fid 62, 63, 64, 65, 66, 89, 90, 91의 user_int_feats_{fid}와 user_dense_feats_{fid}가 모든 행에서 길이 100% 일치 → 전형적 sparse (token_id, weight) 표현. UNI-REC 백본에서는 NS-token으로 들어갈 때 가중합 임베딩 풀링으로 처리.

3.6 dense feature / 시간 분포

sample timestamp는 2026-03-05 13분 구간에 집중. label_time − timestamp는 median 192s, max 832s. 즉 timestamp=노출, label_time=피드백. 학습/평가 split은 반드시 label_time 기준.

3.7 12개 발견 정리

1) Per-row scoring 형식 (next-item 생성이 아님)

1000 unique user ↔ 1000 행, label binary {1,2}, 7.1:1 불균형.

→ pointwise/pairwise scorer. label=2를 양성으로 두고 focal loss 권장.

2) 심한 클래스 불균형 (≈ 7.1 : 1)

→ class-balanced sampling 또는 focal loss. ROC-AUC와 PR-AUC 동시 보고.

3) 매우 긴 행동 시퀀스 (D=1100+)

→ OneTrans의 pyramid query pruning 또는 InterFormer의 bridging arch로 입력 길이를 점진적 축소.

4) 도메인 vocab 거의 disjoint (Jaccard 0.7–10%)

→ 도메인별 임베딩 테이블 + 도메인 separator token. OneTrans 식의 timestamp-aware interleaving merge 적용.

5) Domain C가 target item 공간과 가장 가까움 (11.1%)

→ unified backbone에서도 C-시퀀스 위치 토큰에 더 큰 query budget 할당 (또는 separate head). InterFormer의 Cross Arch로 명시적 cross.

6) Target leakage 거의 없음 (4/1000)

→ 시퀀스 그대로 입력 가능. 모델은 일반화가 핵심.

7) (id, weight) 정렬 페어 발견 (fids 62, 63, 64, 65, 66, 89, 90, 91)

→ NS-token으로 변환할 때 pool(fid) = Σ softmax(weight)_k · Emb(id_k)로 가중합. CAN(WSDM 2022) 스타일.

8) 도메인 내 길이 일관성 100% (튜플 시퀀스)

→ 한 이벤트 = (item_id, category, brand, …) 튜플. 도메인 인코더는 필드별 임베딩 concat → projection → token.

9) label=2 사용자가 약간 더 긴 engagement

→ sequence-length feature를 NS-token으로 추가 (cheap win).

10) timestamp = unix s, label_time ≥ timestamp

→ split은 label_time 기준; (label_time − timestamp) 버킷도 NS-token.

11) Dense feature 정규화 안 됨

→ per-feature log+znorm; 또는 AutoDis(KDD 2021) 학습형 디스크리타이저.

12) item_int_feats_11 (item 측 단일 multi-valued)

→ NS-token으로 mean/attn pool. user 측 태그 풀과 대칭 처리.

4. 통합 백본 전략 (UNI-REC native)

주최측 권장 3편을 우리 데이터에 정확히 매핑한다. 핵심 청사진은 OneTrans를 따르고, 도메인 간 크로스 신호 강화에 InterFormer의 Cross Arch를 차용, sequence/feature 균형 검증은 HyFormer의 ablation 프로토콜을 사용한다.

4.1 OneTrans 청사진 (가장 직접적)

OneTrans는 모든 입력을 한 토큰 시퀀스로 만든다.

X⁽⁰⁾ = [ S-tokens ; NS-tokens ] ∈ R^((Lₛ + Lₙₛ) × d)
  • S-tokens: 행동 시퀀스. 여러 시퀀스를 timestamp-aware interleaving merge 또는 impact-ordered concat (학습형 separator token 사이에 끼워서) 으로 단일 시퀀스화.
  • NS-tokens: user 프로필 / item 속성 / context. Auto-Split (모든 피처 concat → 단일 MLP → Lₙₛ개 토큰으로 split) 이 manual grouping보다 우수.
  • Mixed Causal Attention: S-token은 자신의 과거 S-token만 본다 (autoregressive). NS-token은 전체 S history + 자신 이전의 NS를 모두 본다 (target attention 수렴).
  • Mixed parameterization: S-token은 layer 전체에 Q/K/V·FFN 공유. NS-token은 token-specific 파라미터. (전부 공유했을 때보다 +0.15% UAUC.)
  • Pyramid query pruning: 깊이가 깊어질수록 query 토큰 수를 점차 줄여 (OneTransS: 1190→12, OneTransL: 1500→16), O(L·L′·d) attention으로 비용 감소.
  • RMSNorm pre-norm: token 종류별 통계 차이로 인한 attention collapse 방지.
  • Loss: CTR/CVR head — P(click=1|NS, S; Θ), P(conv=1|click=1, NS, S; Θ). AUC와 UAUC로 평가.

4.2 InterFormer 보완 (양방향 모드 정보 흐름)

OneTrans가 “하나로 합쳐 attention” 이라면, InterFormer는 “모드를 분리해 두고 양방향으로 cross”. 주요 컴포넌트:

  • Interaction Arch — sequence query를 받아 behavior-aware non-sequence embedding 생성.
  • Sequence Arch — non-sequence query를 받아 context-aware sequence embedding 생성.
  • Cross Arch — 위 둘 사이를 bridging하며 양방향 정보 교환. early summarization 회피.

우리 데이터의 4-domain disjoint vocab이라는 특성상, 도메인별 Sequence Arch + 공통 Interaction Arch + 도메인↔공통 Cross Arch 구조가 자연스럽다.

4.3 HyFormer (균형 검증용)

OneTrans는 “seq를 NS에 흡수” 쪽으로, InterFormer는 “양 모드 보존” 쪽으로 기울어진다. HyFormer는 한쪽으로 치우치지 않는 hybrid를 주장한다. 우리는 P2 단계에서 HyFormer의 ablation 프로토콜로 “seq vs feature 가중치”를 sweep해 데이터 친화 균형점을 찾는다.

4.4 우리 데이터로의 매핑 (구체)

┌─── separator tokens ───┐ S-tokens (interleaved): [ A_1, A_2, …, A_9 | sep_A→B | B_1, …, B_14 | sep_B→C | C_1, …, C_12 | sep_C→D | D_1, …, D_10 ] ▲ ▲ DIN-style C에 더 큰 query budget └ 각 도메인 내부: tuple 임베딩 (item_id ⊕ category ⊕ brand …) → MLP → d-dim token NS-tokens (Auto-Split): raw = concat( Embed(35 user scalar int), # one-hot or learned embedding Embed(13 item scalar int), WeightedPool(11 user int-list & 1 item int-list), WeightedPool(8 aligned (id, weight) pairs at fids 62-66, 89-91), # CAN-style Norm(10 user dense), # log+znorm or AutoDis Embed(time-bucket(label_time - timestamp)), Embed(seq_length_per_domain bucket) ) NS-tokens = split(MLP(raw), L_NS) # OneTrans Auto-Split X⁽⁰⁾ = [ S-tokens ; NS-tokens ] For each layer ℓ: Q/K/V = MixedProj(X⁽ℓ⁾) # S: shared params; NS: token-specific X⁽ℓ⁺¹⁾ = MixedFFN(Attn_causal_mixed(Q, K, V) + X⁽ℓ⁾) if ℓ % stride == 0: prune queries (pyramid) Heads: CTR_head( token_NS_target ) → sigmoid → label_type=2 score (옵션) Generative head: Hierarchical SID over candidate items (P3에서)

4.5 단계별 로드맵

단계산출물주요 메트릭스트레치
P0 — Tabular baseline DCN-V2 / GBDT (LightGBM) on NS만. 도메인 시퀀스는 last-N mean-pool로만. label-2 ROC-AUC, PR-AUC LR 대비 +5 pt
P1 — OneTrans-tiny 4-도메인 interleaved S-tokens + Auto-Split NS-tokens, mixed causal attention, pyramid 없이 4-layer. +1.5–2 pt UAUC OneTrans 논문 figure 3 스케일 곡선과 비교
P2 — InterFormer Cross Arch 추가 도메인별 Sequence Arch + 공통 Interaction Arch + 양방향 Cross Arch. HyFormer 프로토콜로 seq/feat 가중치 ablation. +0.5–1 pt 도메인별 contribution 분해 리포트
P3 — Pyramid + scaling OneTransL 스케일(파라미터 ~330M, layer ≥ 16) + KV cache. scaling-law slope 비교 online-style A/B 시뮬레이션
P4 — Generative head (옵션) multimodal item embedding 공개 시 RQ-VAE Semantic ID + InfoNCE retrieval head. HR@10, NDCG@10 OnePiece 권역 도달
왜 이 순서인가 — 주최측 reading list가 명확히 “단일 백본” 쪽을 가리키므로, P0 베이스라인 직후 곧바로 OneTrans-tiny로 진입하는 게 가장 효율적이다. SASRec/BERT4Rec/DIN을 따로 만들어 ensemble하는 전통 경로는 점수 천장이 더 낮을 가능성이 크다.

5. 이런 대회는 보통 어떻게 접근하는가

  1. 정찰 — task / metric / 권장 reading list / 작년 결과를 먼저 읽음. (지금 단계.)
  2. End-to-end 베이스라인을 24h 안에 — LR/LightGBM이라도 좋다. 이게 있어야 모든 변경의 효과를 측정 가능.
  3. Validation을 LB와 일치시키기 — label_time 기준 시간 split. offline ↔ LB 상관 0.8 이상 나올 때까지 split을 손본다.
  4. 가설 → 단일 변경 → ablation → 기록 — 매주 1~2개 변경만. MLflow / W&B 필수.
  5. 복잡도는 phased — P0 → P1 → P2 → … 윗 단계는 아래가 plateau 친 뒤에만.
  6. 마지막 2주는 ensemble + calibration + post-rerank — 단일 SOTA보다 다중 시드 평균이 거의 항상 이긴다.
핵심 원칙 — fast iteration loop. 하루 5번 실험 돌릴 수 있는 인프라가 우승한다.

6. Round 1 시작 시 즉시 확인할 사항

  1. 공식 metric — UAUC인지 GAUC인지 HR@K인지. (OneTrans는 UAUC, 2025 baseline은 retrieval HR@K.)
  2. multimodal item embedding 동봉 여부 — 동봉되면 P4(Semantic ID generative head)가 가장 큰 swing.
  3. 도메인 A/B/C/D의 의미 공식화 — 광고/검색/콘텐츠/앱 매핑이 풀리면 expert routing prior에 직결.
  4. 제출 포맷 — score 파일인지 ranked list인지. 후자면 retrieval head 우선순위 상승.
  5. AngelML 환경 제약 — GPU 시간 / 모델 크기 / 추론 latency cap.
  6. baseline_2026 공개 여부 — github.com/TencentAdvertisingAlgorithmCompetition 와 huggingface.co/TAAC2026 매일 확인.
